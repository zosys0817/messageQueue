# Anti-Patterns (하지 말 것)

## 환경 & 아키텍처

| DO NOT | 대신 |
|--------|------|
| 막연하게 AWS라고 가정 | 항상 AWS/NCP/On-Prem 중 어디인지 확인 |
| 요청 없이 복잡한 아키텍처(MSA 등) 제안 | 확장성 필수 아니면 **Monolithic & Simple** 유지 |
| Over-engineering | "One-Man Army"가 관리 가능한 수준인지 점검 |
| 외부 의존성 무분별 추가 | 새 라이브러리 도입 시 유지보수 비용 먼저 언급 |

---

## 코드 품질

| DO NOT | 대신 |
|--------|------|
| 읽지 않은 코드 수정 | 기존 코드를 먼저 읽고 맥락 파악 후 수정 |
| Quick & Dirty 코드 (경고 없이) | 기술 부채 발생 시 명시적으로 경고 |
| 에러 처리 생략 | **'왜'** 실패했는지 맥락이 담긴 로그 포함 |
| deprecated/EOL 기술 제안 | 활발히 유지보수되는 LTS 버전 우선 |

---

## 배포 & 변경

| DO NOT | 대신 |
|--------|------|
| 검증 없이 프로덕션 배포 제안 | 반드시 dry-run/staging 검증 단계 포함 |
| 한 번에 여러 파일 대규모 변경 | 작은 단위로 쪼개서 단계별 검증 |
